## 2. 实现思路

### 2.1 技术选型

- **Godot 4.6**：2D 竖屏手游上手快，场景化 UI 适合还原参考图三屏结构。
- **分层架构**：数据层（牌型、关卡）→ 生成层（布局、洗牌、可解性）→ 表现层（牌面、HUD、音效），便于分阶段用 AI 辅助实现和排查。
- **OpenSpec**：用 spec-driven 方式记录 P0～P5 各阶段需求与归档，避免「只有代码、没有设计依据」。

### 2.2 AI 工具使用方式

本次开发 **大量使用 Cursor Agent** 辅助，但关键决策与验收仍由人工完成：

| 环节 | AI 辅助 | 人工负责 |
|------|---------|----------|
| 架构与模块拆分 | 生成方案、补全样板代码 | 确认分层边界、Godot 节点结构 |
| 玩法规则 / 可解性 | 实现 `BoardRules`、`BoardSolver` 草稿 | 真机试玩、边界 case |
| UI 场景 | 生成 `.tscn` 骨架与脚本绑定 | **在编辑器里拖布局**、对参考图微调 |
| 资源 | AI 批量生图、自动切图脚本 | **PS 逐张修图**、统一尺寸与命名 |
| 文档 | README、OpenSpec、本总结 | 删减、对齐题目与真实实现 |

原则：**玩法与输入链路必须可玩**；美术先程序化生图/白模保底，再换 PS 修过的 PNG。

### 2.3 核心流程

```
启动 Main.tscn
  → 主页（继续当前解锁关 / 选关 / 设置）
  → 对局（程序化生成关卡 → 点选配对消除）
  → 通关结算（时间 / 分数 / 连击 → 下一关或回主页）
```

20 关不依赖 20 份手写布局 JSON，而是：

- **固定 5×7 槽位**（`GridSlots`）：底层 35 格 + 半格偏移中层 + 顶层；
- **按关卡递增层数与牌数**（`LayoutGenerator` + `LevelCatalog`）；
- **洗牌后做可解性验证**（`BoardSolver`），失败则重试。

进度保存在 `user://vita_progress.json`，通关解锁下一关。

## 3. 功能清单（对照参考图）

### 首页

- 木纹背景、Logo、门环装饰、橙色「关卡 N」大按钮、选关入口、设置按钮。
- 启动播放主界面 BGM（`BgmManager.play_home()`）。

### 游戏页

- 顶栏：返回、关卡 / 分数 / 匹配数、菜单（打开选关）。
- 牌桌：多层叠牌、贴图或程序化牌面、选中动画、消除动画。
- 底栏：洗牌、提示（带剩余次数角标）。
- 点击被挡牌时顶栏下飘字（「该麻将被盖住」「两边被锁住」）。
- 配对成功在碰撞点上方飘 `+320` 分。

### 结果页

- 「才华横溢」标题、时间 / 分数 / 连击统计、鼓励文案、绿色下一关按钮、返回主页。
- 通关后解锁下一关并持久化。

### 选关

- 20 关列表，显示名称、难度星级、锁定状态。
- 已解锁进度提示「已解锁 X / 20 关」。

## 4. 创新与体验优化（题目鼓励项）

1. **ScrollListTap（选关列表）**  
   对标 Android `RecyclerView` touch slop：手势在 `ScrollContainer` 统一处理，关卡行 `mouse_filter=IGNORE`，**滑动与点击分离**，解决「在按钮上滑不动」和「滑动误触进关」。已写入 `openspec/specs/level-progression/spec.md`。

2. **程序化 20 关**  
   槽位 + 层数分布 + 可解性验证，避免手工维护 20 个布局文件，难度随关卡 ID 递增。

3. **场景驱动 UI**  
   布局、贴图、StyleBox 均在 `.tscn` 中编辑；脚本只更新文案与数值，避免运行时拼 UI 导致调位置无效。

4. **音频与设置**  
   主页 / 进关 / 通关三套 BGM；音乐开关与音量写入 `user://vita_settings.json`。

5. **热更新预留**  
   `HotUpdateManager` + PCK 热更管线（作业场景可不启用远程 CDN）。

## 5. 遇到的问题与解决方案

### 5.1 移动端触摸与 UI

| 问题 | 原因 | 解决 |
|------|------|------|
| 点牌无反应 | HUD 触摸区 `STOP` 挡住牌桌 | 对局区 `IGNORE`，`main.gd` 转发到 `Board` |
| 点一次选上又取消 | 触摸模拟鼠标双发 | `PointerInput` 同帧去重 |
| 选关滑动误触进关 | 按钮自己处理 `gui_input` | `ScrollListTap` + 行节点 `IGNORE` |
| 在按钮上滑不动列表 | 子节点 `mouse_filter=STOP` 抢手势 | 同上，手势交给 `ScrollContainer` |

### 5.2 布局与显示

| 问题 | 原因 | 解决 |
|------|------|------|
| 三层牌像三行平铺 | 层间距过大、`y_sort` 盖层 | 收紧 `GRID_Y_STEP`，用 `z_index` 控前后 |
| 超长屏黑边 | `stretch/aspect` 不合适 | 改为 `keep_width`，牌桌 `BoardLayoutScaler` 自适应 |
| 改 `ui_constants` 顶栏不动 | 代码 `_ready` 覆盖场景 | 删除运行时改布局，**只在场景里调** |
| 结算文案竖排 | `Label` 宽度被压成一字宽 + `autowrap` | `size_flags` 改为横向铺满 |

### 5.3 代码与引擎

| 问题 | 现象 | 解决 |
|------|------|------|
| `GridSlots` 找不到 | 脚本里写了 `//` 整除 | GDScript 改为 `/` |
| `button_press_scale` 报错 | 无 meta 时 `get_meta` | 先 `has_meta()` |
| 改场景树后崩溃 | `@onready` 路径未更新 | 节点挪动后同步改脚本路径 |

### 5.4 资源管线（耗时最多）

| 阶段 | 问题 | 处理 |
|------|------|------|
| AI 生图 | 风格不一、字牌糊、小尺寸难认 | 作参考，不全信 |
| 自动切图 | 尺寸不一、命名混乱 | 放弃全自动，改 PS |
| PS 手修 | 34 张牌 + 三套 UI 工作量大 | 统一画布、居中、调色、按 `TEXTURE_MAP` 命名导入 |
| 引擎兜底 | 缺图或坏图 | `TileTextureAtlas` 回退程序化绘制，保证不崩 |

**教训**：手游 UI 不要指望「生一大张 → 自动切 34 张」一步到位；应先定规范再进引擎。

## 6. 已知不足

- **美术**：牌面与参考 Vita 仍有差距，部分牌靠逐张 `scale` 对齐。
- **结算页**：参考图有通关进度条，当前已显示「关卡 X / 20」进度条。
- **连击数**：结算页展示的是本关匹配次数，非连续 combo 链。
- **刘海屏**：已移除代码侧安全区适配，极端机型顶栏可能偏贴边（可在场景里加大 `margin_top`）。
- **热更**：启动会 `await HotUpdateManager`，无 CDN 时仅快速跳过，对作业无影响。

## 7. 如何运行与导出

### 编辑器运行

1. 安装 **Godot 4.6**
2. 打开本项目，主场景 `Main.tscn`
3. **F5** 运行

### Android 导出

1. Godot 安装 Android 导出模板与 SDK/NDK  
2. 打开 **项目 → 导出**，选择预设 **Android**（`export_presets.cfg`）  
3. 导出 APK，安装到手机验证：主页 → 进关 → 消除 → 结算 → 选关滚动

### 关键目录

| 路径 | 说明 |
|------|------|
| `Scenes/` | 三屏 UI 与牌桌场景 |
| `Data/Levels/levels.json` | 20 关配置 |
| `Scripts/ui/scroll_list_tap.gd` | 选关滑动/点击分离 |
| `openspec/specs/` | 能力规格（评审可参考） |
| `README.md` | 更长的开发心得与资源教训 |

## 8. 总结

本项目在 **24 小时量级** 内完成了 Vita Mahjong 核心三屏流程、20 关进度与移动端可玩版本。开发上采用 **AI 辅助编码 + 人工场景与美术修图 + OpenSpec 记录** 的组合；体验上重点解决了 **触摸链路、选关滑动误触、叠层布局** 等真实手游问题。

若继续打磨，优先顺序建议：**结算进度条 → 真机截图与 APK 交付 → 牌面统一尺寸图集**。
