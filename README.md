# Vita Mahjong

Godot 4.6 竖屏麻将消除游戏，设计分辨率 **720×1080**，目标平台 Android。

- 详细技术说明：`docs/project-summary.md`
- **笔试提交总结**：`docs/笔试总结.md`
- UI / 贴图资源说明：`Assets/README.md`
- 总体设计：`openspec/design.md`

## 快速开始

1. 用 Godot 4.6 打开本项目
2. 主场景：`Main.tscn`
3. 按 **F5** 运行

## 开发心得

这份心得记录了我们从零搭到可玩版本过程中踩过的坑、做过的取舍，以及资源管线上的教训。适合以后自己回看，也适合接手的人快速建立预期。

### 1. 先分层，再堆功能

项目按 **数据层 → 生成层 → 表现层** 推进最稳：

| 层级 | 职责 | 代表模块 |
|------|------|----------|
| 数据 | 牌型、格子、关卡配置 | `TileRegistry`、`CellData`、`levels.json` |
| 生成 | 布局、洗牌、可解性、建盘 | `GridSlots`、`LayoutGenerator`、`BoardBuilder` |
| 表现 | 场景、动画、音效、UI | `MahjongTile`、`GameHUD`、`ResultOverlay` |

白模阶段先把「能摆盘、能点、能消」跑通，再换贴图和打磨 UI，避免一边调美术一边改规则，返工成本会很高。

### 2. 叠层布局：看起来不像麻将，多半是间距和排序

早期三层牌像三行平铺，根因有两个：

- **纵向步进太大**：层与层之间按整牌高（102px）排，视觉上没有压叠感。后来收紧 `GRID_Y_STEP`，并给每层加像素偏移。
- **`y_sort_enabled` 开错了**：2D 节点按 Y 排序后，底层反而可能盖住上层。牌桌应靠 `z_index = layer * 100 + grid_y` 控制前后关系，而不是依赖 Y-sort。

另外，布局生成从手写 JSON 演进到 **5×7 固定槽位**（`GridSlots`）：L0 底层 35 格、L1 半格偏移 24 格、L2 再叠 15 格。上层牌必须「坐」在下层两张牌中间，不能随意摆坐标。

### 3. 点击与触摸：事件链路比想象中长

手机上点不到牌，通常不是规则错了，而是 **输入被 UI 吃掉** 或 **坐标没对齐**：

- `ColorRect` / `Label` 默认会拦截鼠标事件，牌节点里要给装饰控件设 `mouse_filter = IGNORE`，只留 `Area2D` 接点击。
- `GameHUD` 的对局触摸区如果设为 `STOP`，会挡住牌桌。改成 `IGNORE`，由 `main.gd` 统一转发到棋盘。
- 触摸在部分机型上会 **连发两次**（选中又立刻取消），`PointerInput` 里做了同帧去重。
- 视口缩放后，要用 canvas 坐标而不是裸屏幕坐标做命中检测。

### 4. 移动端显示：黑边、长屏、安全区

- `stretch/aspect` 从 `expand` 调到 **`keep_width`** 后，超长屏以宽度铺满，减少左右黑边。
- `emulate_mouse_from_touch = true` 让触摸走统一输入路径。
- `ScreenAdapter` 提供安全区 insets，顶栏 / 底栏 / 牌桌居中都要考虑刘海和导航条。
- 牌桌用 `BoardLayoutScaler` 按可视区域自动放大，5×7 布局尽量占满中间游戏区，而不是固定死 73×102 挤在中间。

### 5. GDScript 与 Godot 4 的小坑

这些是写代码时实打实撞过的：

| 问题 | 现象 | 处理 |
|------|------|------|
| 无类型空数组 `[]` | 不能传给 `Array[String]` 参数 | 声明 `const EMPTY: Array[String] = []` |
| `//` 整除 | `GridSlots` 解析失败，连带 `LayoutGenerator` 报「找不到类」 | GDScript 用 `/`，两个 int 相除即整数除 |
| `get_meta()` 未先判断 | `button_press_scale.gd` 运行时报错 | 先 `has_meta()` 再 `get_meta()` |
| 参数名遮蔽成员 | 一堆 Parser/编译警告 | 如 `visible` → `parts_visible`，`category` → `tile_category` |
| 改场景树不改脚本 | `ResultOverlay` 把 `TitleLabel` 挪到 `Frame` 下后，`$TitleLabel` 变 null | 节点路径要跟着场景结构同步改 |

**教训**：Godot 报「找不到某个 `class_name`」，优先打开那个类文件本身看有没有语法错误，不要只在引用方排查。

### 6. UI 与场景编辑：编辑器里改布局，脚本只管数据

三屏结构（主页 / 对局 HUD / 结算）定型后，约定是：

- **位置、锚点、贴图** → 在 `.tscn` 里拖，别在运行时拼。
- **脚本** → 只负责动态文案、数值、信号。

结算页我们来回调过好几次纵向位置（莲花、祥云框、「才华横溢」、统计卡片整体下移）。后来把 `TitleLabel`、`GlowTop` 收进 `Frame` 节点，结构更清晰，但 **`result_overlay.gd` 的 `@onready` 路径必须一起改**，否则就是 `null instance` 赋值崩溃。

按钮反馈统一用 `ButtonPressScale` 做按下缩放，手感会统一很多。

### 7. 音频与设置

- `BgmManager`：`game_running` 主界面循环，`game_start` 进关播一次，`game_win` 通关。
- `GameSettings` 持久化音乐开关和音量到 `user://vita_settings.json`。
- `SfxManager` 用程序化短音（点击、配对、洗牌），不依赖外部音效包，迭代快。

### 8. 热更新（PCK）

`HotUpdateManager` + `load_resource_pack()` 走 Godot 4.6 的 bool 返回值（不是 Godot 3 的 `OK` 错误码）。资源路径仍用 `res://`，打包进 PCK 后会自动覆盖同名资源。适合后期只更 UI / 音频 / 关卡 JSON，不必每次发整包。

### 9. OpenSpec 与文档习惯

我们用 OpenSpec 按 P0～P5 分阶段归档（`openspec/changes/archive/`），主 spec 在 `openspec/specs/`。好处是：

- 每个阶段有 proposal / design / tasks / specs，不会只剩一堆代码没人知道为什么这样写。
- 改完一阶段就归档，避免「活跃 change」堆积。

如果只在聊天里改了代码、没走 change 流程，文档就会滞后——例如结算页布局微调发生在 P5 归档之后，属于 UI 细修，最好以后单独开一个小 change 或直接在 `Assets/README.md` 记一笔。

## 资源管线教训：AI 生图 → 自动切图 → PS 手修

这是本次开发里 **最耗时间、但最不可替代** 的一段。流程实际是三步走，不是「生图完就能进游戏」。

### 生图效果不理想

麻将牌面用 AI 批量生成后，整体问题包括：

- 风格不统一：有的偏写实、有的偏卡通，花色颜色饱和度差很多。
- 牌面识别度不够：万 / 条 / 饼 / 字牌在小尺寸（游戏内约 73×102）下糊成一团。
- 部分字牌、箭牌生成质量明显差于数牌，东南西北经常「长得像但不完全一样」。
- 文件名和内容对不上（例如 `tile_00_01_eewan_1.png` 被映射去当中，`tile_fafafa_wan_1.png` 这类命名也暴露出生图 / 切图流程很随意）。

当前代码里 `TileTextureAtlas` 的策略是：**有映射且文件存在就用 PNG，否则回退到程序化绘制**。所以坏图不会崩，但观感参差。

### 自动切图：尺寸不一、命名混乱

从 Downloads 批量切图放进 `Assets/Mahjong/` 后，实际情况是：

- 单张像素并不统一（设计目标 210×310，实际各张不一致）。
- 同一张图被切成多个碎片，或不同图切出重复区域。
- 命名规则不统一：`tile_28_29_nan.png`、`tile_nanan.png`、`tile_xixix_1.png` 并存，维护 `TEXTURE_MAP` 很痛苦。
- 游戏内靠 `get_sprite_scale()` 按每张图宽高单独缩放，能跑，但 **牌与牌之间细微大小差** 一眼能看出，破坏整齐感。

### PS 手修：自动流程补不上的，只能一张张抠

AI 生图 + 自动切图走不通之后，**大量资源是自己在 Photoshop 里逐张修的**。这也是目前游戏里 UI 和部分牌面能看的主要原因——代码只能缩放和映射，救不了源头素材的参差。

实际手修时反复在做这些事：

| 修什么 | 常见操作 | 为什么必须手修 |
|--------|----------|----------------|
| 麻将牌面 | 裁到统一画布、对齐牌面居中、修边缘毛边、调色统一 | 自动切出来的单牌尺寸不一，有的偏左有的留白过多 |
| 字牌 / 风牌 | 重写字感、统一笔画粗细、删掉 AI 乱画的噪点 | 这类牌 AI 最差，缩到 73×102 后几乎没法认 |
| UI 按钮 / 边框 | 抠透明底、修锯齿、拉正变形、补缺失像素 | 切图工具按块切，按钮圆角和描边经常被切残 |
| 主页 / 结算装饰 | 拼贴、去底、调亮度对比、对齐设计稿 | 一张大图切成多个 Part 后，接缝和色差不手修很明显 |
| 命名与导出 | 按 `TEXTURE_MAP` 或 `Assets/README.md` 映射重命名，统一 PNG 导出 | 源文件名是 `Image_10_Part_3` 这类机器名，不进引擎 |

手修后的感受：

- **比写代码还慢**：34 张牌 + 三套 UI，一张张对齐尺寸，一晚上可能只搞定一个界面。
- **但质量跳变明显**：统一成相同宽高（例如 210×310 或约定好的单一尺寸）后，牌桌整齐度立刻好很多，不再靠每张图单独 `scale` 硬撑。
- **适合「先跑通再换皮」**：程序化生图 / 白模保证玩法不断，PS 修好的图覆盖进 `Assets/`，改 `TEXTURE_MAP` 或直接在场景里换 Texture 即可，不用动逻辑。
- **最好留 PSD 分层**：后面要改配色、加阴影、出 2x 高清包，有源文件比只有导出 PNG 省事得多。

源文件习惯放在本机 Downloads 的设计目录（见 `Assets/README.md` 里的原始路径），修完再复制进 `Assets/Mahjong/`、`Assets/UI/` 各子目录，Godot 重新导入生成 `.import`。

### 如果重来，我会这样做

1. **不要指望「生一大张 → 自动切 34 张」一步到位。** 要么买/找一套统一风格的麻将 sprite sheet，要么固定单牌画布尺寸（例如统一 256×360）逐张生成，**并预留 PS 批量动作**（统一画布、居中、导出）。
2. **先定规范再进引擎**：统一尺寸、统一透明边距、统一命名（`wan_1.png` … `dragon_white.png`），`TEXTURE_MAP` 只做 id → 文件名，不做补救性映射。
3. **用图集（Atlas）或一张大图**，在 Godot 里 `AtlasTexture` 切区域，比文件夹里 34 个大小不一的 PNG 好管得多；PS 里也可以先修一张规范单牌，再当模板批量套。
4. **验收标准**：缩到 73×102 后，相邻两张同花色仍清晰可辨；四张风牌轮廓一致；字牌笔画不糊。
5. **短期兜底**：程序化生图（`TileTextureAtlas._build_procedural_texture`）保证玩法；**中期以 PS 手修 PNG 为准**；长期再考虑统一外包或购买成套素材一次性替换。
